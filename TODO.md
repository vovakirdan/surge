# Большой и подробный план разработки

## Фаза A — Бутстрап фронтенда

**Цель:** надёжный лексер/парсер, AST, диагностика.

1. **Лексер**

   * Токены: идентификаторы/ключевые слова, литералы (int/float/string), операторы `+ - * / == != < <= > >= = :=`, скобки/знаки, комментарии `//` и `/*…*/`.
   * Позиции (line/col), сообщения об ошибках.
   * **DoD:** прогон `tests/lexing` с золотыми эталонами.

2. **Парсер (LL с рекурсивным спуском)**

   * Мини-грамматика: `let`, `signal`, `fn`, `if/else`, `while`, массивы, вызовы, `parallel map/reduce`, `:=` (реактивное связывание).
   * Приоритеты операторов, унарные/бинарные.
   * AST-узлы с явными типами нод.
   * **DoD:** `tests/parsing` — сравнение деревьев/«снимков» (snapshot tests).

3. **Диагностика**

   * Формат: `file:line:col: error: message`.
   * Контекст строки с указателем `^`.
   * **DoD:** 10+ негативных кейсов.

> **Решения:** без генераторов парсеров (ручной), чтобы легче эволюционировать синтаксис.

---

## Фаза B — Типы и семантика

**Цель:** статическая проверка, строгие типы.

1. **Типы**: `int`, `float`, `bool`, `string`, `T[]`, `channel<T>`, `&T` (borrow), `own T` (owning).
2. **Символьные таблицы, области видимости**, запрет теней (или controlled).
3. **Проверка «чистоты» функций**

   * Аннотация `@pure` (на MVP — белый список рантайм-функций).
   * Нечистым (I/O, глобалки) запрещено в `parallel` и реактивных вычислениях.
4. **DoD:** `tests/sema` — положительные/отрицательные, включая ошибки типов.

> **Решения:** без автоматического вывода типов вначале; явно указывать типы параметров и возвращаемых значений.

---

## Фаза C — Дизайн байткода и формата .sbc

**Цель:** стабильный минимальный ISA и упаковка артефактов.

1. **Стековая VM**

   * 32/64-битные целые (выбрать `int64_t`), `double` для float.
   * Кадр вызова: локалы, указатели на константы/код.
2. **Опкоды (минимум)**

   * Константы: `PUSH_I64`, `PUSH_F64`, `PUSH_STR`, `PUSH_BOOL`, `PUSH_NULL`
   * Арифметика: `ADD`, `SUB`, `MUL`, `DIV`
   * Сравнения/ветвления: `CMP_EQ/NE/LT/LE/GT/GE`, `JMP`, `JMP_IF_TRUE/FALSE`
   * Локалы/глобалы: `LOAD`, `STORE`, `GLOAD`, `GSTORE`
   * Вызовы: `CALL`, `RET`
   * Масивы/индексация: `ARR_NEW`, `ARR_LEN`, `ARR_GET`, `ARR_SET`
   * Спец.: `HALT`
3. **Формат .sbc**

   * Заголовок (магическое число `SBC0`, версия).
   * Констант-пул (строки, числа), таблица функций (адреса, сигнатуры), секция байткода.
4. **Дизассемблер**

   * Чтение `.sbc`, печать опкодов, констант, таблиц.
5. **DoD:** документ `docs/BYTECODE.md`, дизассемблер показывает ожидаемое.

## Фаза D — VM и память

**Цель:** рабочая VM + простая и быстрая модель памяти.

1. **VM-цикл**

   * Декодер опкодов, стек, фреймы.
   * Ошибки VM с нормальными сообщениями.
2. **Память**

   * **Арены** (region allocator): короткоживущие объекты (например, запросы HTTP).
   * **RC-heap**: для строк/массивов/замыканий (MVP без детектора циклов).
   * Interning строк (хеш-таблица).
3. **DoD:** `tests/vm` арифметика/функции/массивы/строки.

> **Решения:** сначала арены + RC, потом (позже) лёгкий детектор циклов.

## Фаза E — Компилятор AST → SBC

**Цель:** компилируем выражения/операторы/функции.

1. **Кодоген**

   * Таблицы локалов, диспетч имен.
   * Ветвления, циклы, массивы, вызовы.
   * Оптимизации на проходе: констант-фолдинг, удаление мёртвых ветвей (минимально).
2. **Связка `surgec`**

   * Вход: `.sg` → выход `.sbc`.
3. **DoD:** рядом с VM-тестами выполняем тот же набор через компилятор.

## Фаза F — std::math и базовый рантайм

**Цель:** «математика из коробки» + базовые утилиты.

1. **std\_math**: `sqrt`, `pow`, `sin`, `cos`, `tan`, `exp`, `log`, `min`, `max`, `abs`, `clamp`.
2. **Вызов нативных функций из VM** (таблица builtin-ов, маршалинг аргументов).
3. **I/O минимально**: `print`, `println`, `time.now()`.
4. **DoD:** `examples/math_parallel.sg` и doctest на корректность.

## Фаза G — Реактивность (signals)

**Цель:** привязки `:=`, граф зависимостей, авто-пересчёт.

1. **Модель**

   * `signal x = expr` — узел-источник; `y := expr(x,...)` — вычислимый узел.
   * Точная фиксация семантики `:=`: лениво vs жадно? (MVP: **жадно** при изменении входа).
   * Детекция циклов (ошибка времени выполнения).
2. **Пропагация**

   * Топологический порядок, батчи обновлений.
   * Кэширование значений, инвалидация на изменении.
3. **Параллельность**

   * Вычисление зависимых сабграфов в пуле потоков при чистых функциях.
4. **DoD:** `tests/signals` (цепочки/бродкаст/ветвления/циклы), бенч на N узлов.

---

## Фаза H — Пул задач, futures, `parallel` примитивы

**Цель:** безопасные параллельные вычисления «по умолчанию».

1. **Планировщик**

   * `pthread` пул, work-stealing деки на поток.
   * `future<T>` и `join`.
2. **`parallel map/reduce`**

   * Map: применяет `@pure fn` к массиву, возвращает массив/фьючи.
   * Reduce: безопасная ассоциативная операция (проверка арности/чистоты).
3. **Контракты**

   * В рантайме проверять «чистоту» (запрещать доступ к глобалкам/IO).
   * Аннотация `@pure` и whitelist встроенных.
4. **DoD:** `tests/parallel` — корректность и отсутствие гонок/крашей.

---

## Фаза I — Встроенный HTTP

**Цель:** микрофреймворк «из коробки» для CPU-нагрузки на запрос.

1. **HTTP сервер**

   * `epoll` non-blocking accept/read/write.
   * Роутер: `http.get(path, handler)` / `http.post(...)`.
   * Request/Response: методы, заголовки, тело (ограничить размер на MVP).
2. **Интеграция с планировщиком**

   * Каждый запрос → задача + **своя арена** (авто-free после ответа).
   * Внутри обработчика — `parallel map/reduce` для тяжёлых кусков.
3. **JSON**

   * Мини-парсер/энкодер (векторный рост буфера).
4. **DoD:** `examples/http_sum.sg`, нагрузочный «дымовой» тест (wrk/ab).

**Цель:** ускорить hot-path без усложнения дизайна.

# Важные инженерные решения (кратко)

* **Арены + RC**: дешёвая и понятная модель памяти, совпадает с твоей интуицией из C/Go.
* **Чистота функций**: ключ к безопасной параллельности и реактивности (простая проверка на старте).
* **HTTP = задачи + арены**: каждая ручка чисто изолирована, без явных `free`.
* **Стековая VM**: быстрее в разработке и достаточно шустра для MVP; JIT можно навесить потом.
* **Файл-формат .sbc**: отделяет front-end и back-end, открывает путь к «surge as a library».

# Мини-пример целевого UX (как это будет выглядеть)

`examples/http_sum.sg`

```sg
import http
import json
import math

@pure
fn pow5(x:int) -> int { pow(x, 5) }

fn sum_handler(req: Request) -> Response {
  let xs:int[] = json.parse<int[]>(req.body)
  ys := parallel map pow5 xs     // tasks on thread pool
  total := parallel reduce ys with (acc:int, v:int) => acc + v
  return json.ok({ "sum": total })
}

fn main() {
  http.get("/sum", sum_handler)
  http.listen(":8080")
}
```

## Пример целей по качеству (Definition of Done суммарно)

* Все `tests/*` зелёные (lexer, parser, sema, vm, signals, parallel, http, doctest).
* `examples/*` запускаются через `surge` и `surgec + surge`.
* Без утечек под `valgrind`/`asan`.
* Базовые бенчмарки: `parallel map/reduce` показывает масштабирование по потокам.
* HTTP выдерживает стабильную RPS-нагрузку в «дымовом» профиле и корректно освобождает арены на запрос.
