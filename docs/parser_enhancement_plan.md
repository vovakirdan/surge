# План доработки парсера языка Surge

## Анализ текущего состояния

После анализа кодовой базы парсера (`crates/parser/`) и спецификации языка (`docs/LANGUAGE.md`) выявлено следующее состояние:

### ✅ Уже реализовано

#### Выражения и операторы
- Базовые литералы (int, float, string, bool, nothing)
- Арифметические операторы (`+`, `-`, `*`, `/`, `%`)
- Сравнения (`<`, `<=`, `>`, `>=`, `==`, `!=`, `is`)
- Логические операторы (`&&`, `||`, `!`)
- Битовые операторы (`&`, `|`, `^`, `<<`, `>>`)
- Тернарный оператор (`? :`)
- Оператор нулевого слияния (`??`)
- Присваивания и составные присваивания (`=`, `+=`, `-=`, etc.)
- Индексация массивов (`[]`)
- Вызовы функций
- Унарные операторы (`+x`, `-x`)

#### Операторы управления
- `let` объявления с поддержкой `mut`
- `if`/`else` условные операторы
- `while` циклы
- `for` циклы (C-style и for-in)
- `return` операторы
- `break`/`continue` операторы
- `signal` для реактивных привязок

#### Функции
- Объявления функций с параметрами и возвращаемыми типами
- Атрибуты функций (`@pure`, `@overload`, `@override`, etc.)
- Поддержка generic параметров

#### Шаблоны (patterns)
- Базовые шаблоны для `compare` выражений
- Поддержка tagged constructors
- Шаблоны с условиями (`x if x is int`)

#### Параллельные конструкции
- `parallel map` и `parallel reduce`
- Проверка чистоты функций в параллельных контекстах

#### Типы
- Базовые типы (`int`, `uint`, `float`, `bool`, `string`)
- Ссылочные типы (`&T`, `&mut T`, `own T`)
- Массивы (`T[]`)
- TypeNode структура для представления типов

#### Диагностика ошибок
- Базовая система диагностики с кодами ошибок
- Восстановление после ошибок синтаксиса

## Риски и зависимости

### Риски
1. **Сложность интеграции** - новые конструкции должны правильно интегрироваться с существующей системой типов и диагностик
2. **Производительность** - добавление новых конструкций не должно значительно влиять на скорость парсинга
3. **Совместимость** - изменения не должны ломать существующий функционал

### Зависимости
1. **Лексер** - парсер зависит от корректной токенизации всех новых ключевых слов и операторов
2. **Система диагностик** - нужна интеграция с `surge-diagnostics` для новых кодов ошибок
3. **Тесты** - необходимы как положительные, так и негативные тестовые примеры

## Процесс разработки

1. **Анализ** - изучить спецификацию и существующий код
2. **Планирование** - создать детальный план с чекбоксами
3. **Реализация** - пошаговая реализация новых возможностей
4. **Тестирование** - валидация на примерах из спецификации
5. **Документирование** - обновить документацию по мере реализации

Этот план будет обновляться по мере продвижения разработки и обнаружения новых требований.
