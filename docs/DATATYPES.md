# 1) Базовая модель типов (MVP → дальше)

## Базовые числовые и логические

* `bool` (литералы: `true`/`false` — да, токены уже есть; в парсере добавим как `Primary`).
* Целые: `i8, i16, i32, i64 (alias: int)`, беззнаковые: `u8, u16, u32, u64`.
* Вещественные: `f32, f64 (alias: float)`.
* Строка: `string` (иммутабельная, interning).
* Массив: `T[]` (динамический массив).
* Канал: `channel<T>` (для параллелизма).
* Указатели: `&T` (borrowed), `own T` (owning), `box T` (heap-объект под RC; синтаксис можно включить позже).

Литералы с суффиксами (позже): `123i8`, `1.0f32`. Пока проще: тип вытекает из контекста.

## Статические пользовательские типы (value types)

```sg
type Vec2 { x: f64, y: f64 }      // плоская C-like раскладка, без скрытого заголовка
```

* Это **value type**: копируется как POD, легко класть в массив, передавать по значению.
* Никакого скрытого «объектного» заголовка ⇒ дешёвые аллокации/копии.
* Методы — через `impl` (см. ниже) и/или «extension methods».

## Ссылочные объекты (ref types) — позже

```sg
ref type Image { width: i32, height: i32, data: &u8 }  // хранится на heap/в арене
```

* Имеют «шапку» (vtable ptr/счётчик/флаги) только если **нужна динамика**.
* Создание через `box Image{...}` или арены рантайма (для HTTP-запросов).
* В MVP можно обойтись без `ref type`, стартуя только с value types.

# 2) Объектная модель: «гибкость как в Python, цена как в Go»

## Traits / Protocols (номинальные интерфейсы)

В духе Rust/Go-интерфейсов, но проще:

```sg
trait Add<Rhs, Out> {
  @pure fn add(self, rhs: Rhs) -> Out
}

type Vec2 { x: f64, y: f64 }

impl Add<Vec2, Vec2> for Vec2 {
  @pure fn add(self, rhs: Vec2) -> Vec2 {
    Vec2{ x: self.x + rhs.x, y: self.y + rhs.y }
  }
}
```

* Компилятор сопоставляет операцию `+` с `Add::add` (см. «операторы» ниже).
* Если переменная монотипна (`Vec2`), генерим **статический вызов** (дёшево).
* Если переменная имеет тип «trait объект» (`&dyn Add<...>`), используем **fat-pointer** (ptr + vtable) ⇒ одна индирекция, как в Go.

## Extension methods (расширения без наследования)

```sg
extend int {
  @pure fn clamp(self, lo:int, hi:int) -> int {
    if self < lo { lo } else if self > hi { hi } else { self }
  }
}

let x = 127
print(x.clamp(0, 100))  // 100
```

* Сахар над свободными функциями + резолвинг по типу слева.
* Ноль накладных расходов при статической типизации.

## Операторы как «метаметоды» (но статично разрешаемые)

Связываем операторы с трейтами (фиксированный набор, читаемый для всех):

| Оператор  | Трейт             | Метод   |
| --------- | ----------------- | ------- |
| `+`       | `Add<Rhs, Out>`   | `add`   |
| `-`       | `Sub<Rhs, Out>`   | `sub`   |
| `*`       | `Mul<Rhs, Out>`   | `mul`   |
| `/`       | `Div<Rhs, Out>`   | `div`   |
| `%`       | `Rem<Rhs, Out>`   | `rem`   |
| `==`      | `Eq<Rhs>`         | `eq`    |
| `!=`      | `Eq<Rhs>`         | `!eq`   |
| `<`..`>=` | `Ord<Rhs>`        | `cmp`   |
| `[]`      | `Index<Idx, Out>` | `index` |
| `()`      | `Call<Args, Out>` | `call`  |

Пример индексации:

```sg
trait Index<Idx, Out> { @pure fn index(self, idx: Idx) -> Out }

impl Index<int, f64> for Vec2 {
  @pure fn index(self, i:int) -> f64 { if i==0 { self.x } else { self.y } }
}

let v = Vec2{1.0, 2.0}
print(v[0])  // 1.0
```

**Важно:** мы **не даём определять новые операторы** (чтобы сохранить читаемость и простоту лексера/парсера), но даём переопределять поведение существующих через `impl`. Это даёт «python-like» свободу (как `__add__`), но со **статической ценой** (как в Go/Rust).

## Статическое vs динамическое связывание

* Если тип известен — **инлайн/моно-связывание** (дёшево).
* Если переменная хранится как `&dyn Trait` — **vtable-вызов** (1 индирекция).
* Трейт-объект — **fat-pointer** (как в Rust/Go), ABI фиксируем в `docs/BYTECODE.md`.

# 3) «Динамика» как в Python — но контролируемая и дешевая

## Тип `any`/`variant` (суммарный тип с тегом)

Для мест, где нужна динамика (JSON, REPL, границы HTTP):

```sg
variant Value = Int(i64) | Float(f64) | Bool(bool) | Str(string) | Arr(Value[]) | Obj(Map<string,Value>) | Null
```

* В рантайме — **tagged union** (два машинных слова: тег + payload/ptr).
* Боксинг примитивов по требованию (`box` при поднятии в `Value`).
* Никакой «магии Python» по умолчанию везде! Динамика — **по месту**.

## Принудительный контроль размера числа

Позволяем писать:

```sg
let x:i8 = 120        // статический контроль диапазона при присваивании/литерале
let y:int = x         // неявное расширение допустимо (i8->int)
let z:i8 = y          // запрещено без явного кастинга: `z = (i8)y`
```

* Это даёт «жёсткость C/Go», но при желании — можно «поднять» в `Value`:

```sg
let v:Value = x   // box + тег, динамика дальше решает что делать
```

## Автобоксинг при апкасте (опционально, позже)

* При использовании `any` или `&dyn Trait` возможен автобоксинг, **только если это явно включено** (чтобы не удивлять затратами).
* На MVP — **только явный** `box`/`as any`.

# 4) Память и владение (вписываемся в нашу модель)

* `&T` — заимствование, живёт не дольше области/арены (проверяем на уровне семантики).
* `own T` — владение, перемещение, освобождение при выходе из области/арены (или RC-heap, если `box`).
* `box T` — выделение на heap с RC (и **без** детектора циклов на MVP).
* Для HTTP: каждая ручка/запрос = **своя арена** ⇒ указатели `&T` безопасны и бесплатны; всё освобождается пачкой по окончании запроса.

# 5) Параллельность и «чистота» для методов

* Любая функция/метод может пометиться `@pure`. Тогда:

  * Разрешён в `parallel map/reduce` и внутри реактивных графов.
  * Никакого доступа к глобальному изменяемому состоянию/IO.
* Для **value types** чистота даётся «бесплатно» чаще.
* Для **ref types** — проверяем, что метод не трогает разделяемые мутабельные поля.

# 6) Как это ляжет на компилятор/VM

* **Value types** (POD) → прямой layout в стековых кадрах/массивах.
* **Trait-вызовы**: если переменная как `&dyn Trait`, в байткоде будет опкод `VCALL slot` (вызов по vtable-слоту).
* **Impl-вызовы**: если тип известен → обычный `CALL` на конкретную функцию.
* **Операторы**: транслируются в вызовы трейт-методов. Для примитивов — прямые опкоды (`ADD_I64`, `ADD_F64`).

# 7) План внедрения (минимальными шагами)

**Phase A → B (семантика/типы):**

1. AST: добавить узлы `BoolLit`, `StructType`, `StructLit`, `FieldAccess`, `TraitDecl`, `ImplDecl`, `MethodCall`, `IndexExpr`.
2. Парсер:

   * `type` декларации (`type Name { ... }`)
   * `trait` и `impl` минимально.
   * Литералы `true/false`.
3. Sema:

   * Симтаб для типов/трейтов; проверка метод-резолвинга.
   * Простое сопоставление операторов с трейтом.
4. Байткод:

   * `CALL`, `VCALL`; добавить примитивные арифм. опкоды; схему vtable.
5. Runtime:

   * Таблица vtable для каждого `impl` (фиксируем ABI).
   * Поддержка `Value` (tagged union) — **можно позже** или начать с JSON.

## Мини-кусочки синтаксиса (как это будет выглядеть)

```sg
// 1) value-type + impl оператора
type Vec2 { x:f64, y:f64 }

trait Add<Rhs, Out> { @pure fn add(self, rhs:Rhs) -> Out }

impl Add<Vec2, Vec2> for Vec2 {
  @pure fn add(self, rhs:Vec2) -> Vec2 {
    Vec2{ x: self.x + rhs.x, y: self.y + rhs.y }
  }
}

fn main() {
  let a = Vec2{1.0, 2.0}
  let b = Vec2{3.0, 4.0}
  let c = a + b              // статически резолвится в Vec2::add
  print(c.x, c.y)
}

// 2) extension method
extend string {
  @pure fn upper(self) -> string { to_upper(self) }   // builtin
}

print("hi".upper())

// 3) контролируемая динамика
variant Value = Int(i64) | Float(f64) | Str(string) | Bool(bool) | Null
fn show(v:Value) {
  match v {
    Int(n)   => print("i:", n)
    Float(x) => print("f:", x)
    Str(s)   => print("s:", s)
    _        => print("other")
  }
}
```
