# Phase B — Типы и семантика

Эта директория содержит примеры для тестирования и демонстрации **Фазы B** разработки языка Surge — системы типов и семантического анализа.

## Цели Фазы B

Согласно `TODO.md`, Фаза B должна реализовать:

1. **Строгие статические типы**: `int`, `float`, `bool`, `string`, `T[]`, `channel<T>`, `&T` (borrow), `own T` (owning)
2. **Символьные таблицы и области видимости** с запретом теней
3. **Проверку чистоты функций** для безопасных параллельных операций
4. **Семантические проверки**: соответствие типов, корректность вызовов, валидация операций

## Примеры в этой директории

### 01_basic_types.sg
**Демонстрирует**: базовые типы данных и проверку типов переменных
- Объявление переменных с явными типами (`int`, `float`, `bool`, `string`)
- Переприсваивание с совместимыми типами
- Арифметические операции и сравнения
- **Проверяет**: семантический анализатор должен принимать корректные типы и отклонять несовместимые

### 02_functions_typing.sg
**Демонстрирует**: проверку типов функций и их параметров
- Функции с явными типами параметров и возвращаемых значений
- Вызовы функций с проверкой соответствия типов
- Функции с несколькими параметрами разных типов
- **Проверяет**: корректность сигнатур функций и типов при вызовах

### 03_arrays_indexing.sg
**Демонстрирует**: работу с массивами и типизацию элементов
- Объявление типизированных массивов (`int[]`, `float[]`, `bool[]`)
- Индексация и изменение элементов
- Функции, принимающие массивы как параметры
- **Проверяет**: однородность типов элементов, корректность индексации, типы при присваивании

### 04_signals_reactive.sg
**Демонстрирует**: реактивные сигналы и связывания
- Объявление сигналов с типами (`signal temperature: int`)
- Реактивные связывания с оператором `:=`
- Автоматический перерасчет при изменении сигналов
- **Проверяет**: корректность типов в реактивных выражениях, ограничения на сигналы

### 05_purity_parallel.sg
**Демонстрирует**: проверку чистоты функций для параллельных операций
- Функции с аннотацией `@pure`
- Параллельные операции `parallel map` и `parallel reduce`
- Ограничения на использование I/O и глобального состояния в чистых функциях
- **Проверяет**: только чистые функции могут использоваться в параллельных операциях

## Как использовать

Эти примеры предназначены для:

1. **Разработки семантического анализатора** — каждый файл демонстрирует конкретные аспекты проверки типов
2. **Тестирования** — можно использовать как положительные тесты (должны проходить без ошибок)
3. **Документации** — показывают ожидаемое поведение системы типов

## Связь с существующими тестами

Примеры основаны на тестах из `tests/sema/`:
- `ok/` — положительные случаи (корректный код)
- `bad/` — отрицательные случаи (код с ошибками типов)
- `golden/` — эталонные выходы семантического анализатора

## Запуск

После реализации семантического анализатора в `src/front/sema.c`, эти примеры можно будет тестировать командой:

```bash
make test  # запустит doctests во всех Phase* директориях
```

Или индивидуально через инструменты компиляции:
```bash
build/bin/surgec PhaseB/01_basic_types.sg  # должен пройти без ошибок семантики
```
