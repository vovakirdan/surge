# Что делаем в Phase D (VM и память)

## Цели (итог фазы)

* Рабочая стековая VM, способная исполнять `.sbc` из Phase C end-to-end.
* Простая модель памяти: сочетание арен для короткоживущих объектов и
  reference counting для долгоживущих структур (строки/массивы/замыкания).
* Минимальный набор рантайм-тестов (`tests/vm`) на арифметику, вызовы,
  ветвления, массивы и строки.

## Объем первой версии

* Поддерживаем все опкоды ISA v0: литералы, арифметика, сравнения, переходы,
  вызовы, TRAP/ARR_*.
* Гарантируем корректную работу `CALL/RET` с произвольной глубиной стека и
  локальными переменными.
* В памяти реализуем:
  * арену (region allocator) для временных структур и VM-стека;
  * RC-heap для строк и массивов (без детектора циклов);
  * interning строк констант из `.sbc` (хеш-таблица).
* Глобалы — статический массив `Value gvars[]`, инициализируемый вызовом
  функции `__init__` при старте VM.

## Архитектура VM

1. **Интерпретаторный цикл**
   * Декодер опкодов: таблица с указателями на обработчики.
   * Регистры: `ip`, `bp`, `sp`, `frame`.
   * Проверки границ стека, корректное завершение на `HALT`.
   * Отчёт о ошибках с печатью trap-кода, ip и имени функции.

2. **Кадры вызова**
   * Структура `CallFrame`: `func_desc`, `return_ip`, `base`, `argc`.
   * Параметры располагаются перед локалами; инициализация локалов `null`.
   * `CALL` подготавливает новый кадр, `RET` возвращает управление и значение.

3. **Память**
   * **Arena**: блочный аллокатор из крупных чанков; используется для VM-стека,
     временных массивов Value и вспомогательных структур.
   * **RC-heap**: структурки `RtString`, `RtArray` c счётчиком ссылок;
     освобождение при уходе счётчика в 0.
   * Interning строк: hash map `const_idx -> RtString*` для переиспользования.
   * Заглушки для будущего детектора циклов (Phase E/F).

4. **Рантайм-библиотека**
   * Минимальные helpers: аллокация/деструкция строк и массивов, проверка
     индексов, конвертация в `Value`.
   * TRAP-хэндлер, формирующий человекочитаемое сообщение.

## План работ

1. **Инфраструктура памяти**
   * Реализовать арены (`rt/arena.c`) + unit-тесты.
   * Реализовать RC-heap с API `rt_string_new`, `rt_array_new`.
   * Добавить счётчики ссылок к `Value` (retain/release helpers).

2. **VM-цикл**
   * Таблица обработчиков для опкодов + декодер.
   * Выполнение литералов, арифметики и сравнений.
   * Обработка `JMP*`, включая проверку диапазона.
   * `CALL/RET` + кадры вызова, стек локалов.
   * TRAP-ы и graceful shutdown (`HALT`).

3. **Массивы и строки**
   * Реализация `ARR_NEW/GET/SET/LEN` через RC-heap.
   * Поддержка строковых констант с интернингом при загрузке `.sbc`.

4. **I/O и отладка**
   * Чёткие сообщения об ошибках (ip, функция, trap-code, комментарий).
   * Флаги CLI: `surge runbc <file.sbc>`, опционально `--trace` для дампа
     инструкций (dev only).

5. **Тестирование**
   * `tests/vm` — сценарии исполнения: арифметика, условия, циклы, вызовы,
     работа с массивами/строками, негативные TRAP-кейсы.
   * Интеграция в `make test`.

## Критерии готовности (DoD)

* `surge runbc PhaseC/*.sbc` успешно исполняет примеры.
* Все тесты в `tests/vm` проходят на CI.
* Документация (`docs/BYTECODE.md`, `docs/RUNTIME.md`) обновлена.
* Нет «серых» утечек — RC освобождает строки/массивы после завершения выполнения.

## Риски и смягчение

* **Рост стека**: добавить лимит и trap `STACK_OVERFLOW` (в резерве).
* **Производительность RC**: предусмотреть pooled-allocator для частых
  аллокаций массивов.
* **Ошибки подсчёта ссылок**: включить режим диагностики с логгированием retain/release.
